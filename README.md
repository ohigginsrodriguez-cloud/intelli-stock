# IntelliStock

Plataforma digital que analiza ventas, alerta sobre stock crítico, y recomienda decisiones de inventario para PYMEs. Backend Django 5.2 + DRF, frontend Next.js 16 + Tailwind CSS, base de datos PostgreSQL.

---

## Stack Tecnológico

| Capa | Tecnología | Versión |
|------|-----------|---------|
| Backend | Django + Django REST Framework | 5.2.14 |
| Base de Datos | PostgreSQL | 16 |
| Frontend | Next.js + TypeScript + Tailwind CSS | 16.2.6 |
| Autenticación | JWT (SimpleJWT) | tokens 12h / refresh 30d |
| Alertas | Reglas en Python (`alerts.py`) | — |
| Predicciones | Promedio móvil simple (`predictions.py`) | — |

**Filosofía:** Todo vive en una sola app Django (`inventario`). Sin microservicios, sin ML externo, sin complejidad innecesaria. 11 modelos, un ViewSet por entidad, vistas function-based para reportes y exportación.

### Así se ve en PostgreSQL

Cuando listas las tablas con `\dt` en psql, ves **20 tablas**. No te asustes — 11 son nuestras y 9 son de Django.

**Nuestras (11)** — nombres SIN prefijo `inventario_` porque cada modelo tiene `db_table = 'nombre'`:

```
alerta              → alertas automáticas
compra              → cabeceras de compra
detalle_compra      → productos comprados
detalle_venta       → productos vendidos
negocio             → negocios registrados
producto            → catálogo de productos
proveedor           → proveedores
recomendacion       → recomendaciones inteligentes
suscripcion         → planes de suscripción
usuario             → usuarios (login por correo)
venta               → cabeceras de venta
```

**De Django internals (9)** — las crea automáticamente Django, NO las toques:

```
auth_group                     ← grupos de permisos del admin
auth_group_permissions         ← permisos asignados a grupos
auth_permission                ← catálogo de permisos (uno por modelo+acción)
django_admin_log               ← historial de cambios hechos desde /admin/
django_content_type            ← registro de todos los modelos instalados
django_migrations              ← control de qué migraciones se han aplicado
django_session                 ← sesiones de usuario (no usamos, usamos JWT)
usuario_groups                 ← grupos del usuario (herencia de AbstractUser)
usuario_user_permissions       ← permisos directos del usuario
```

La tabla `usuario` es la nuestra, pero `usuario_groups` y `usuario_user_permissions` son tablas intermedias que Django crea automáticamente porque `Usuario` hereda de `AbstractUser`.

**¿Por qué `db_table` en vez del nombre automático `inventario_producto`?**
Porque el diseño original (`bd_sugerida.txt`) define los nombres en español (`negocio`, `producto`, `venta`, etc.). Django por defecto nombra las tablas como `{app}_{modelo}` (ej: `inventario_producto`), pero al poner `db_table = 'producto'` en el Meta del modelo, forzamos el nombre exacto del diseño original.

### Los `IDENTITY` columns son normales de PostgreSQL

Cuando ves el DDL de una tabla en DBeaver, aparece algo como:

```sql
ALTER TABLE public.producto ALTER COLUMN id_producto ADD GENERATED BY DEFAULT AS IDENTITY (
    SEQUENCE NAME public.producto_id_producto_seq
    START WITH 1
    INCREMENT BY 1
    NO MINVALUE
    NO MAXVALUE
    CACHE 1
);
```

Todo eso (`INCREMENT BY 1`, `NO MINVALUE`, `NO MAXVALUE`, `CACHE 1`) lo pone **PostgreSQL automáticamente**. Es el estándar moderno desde PostgreSQL 10 para crear columnas auto-incrementables (reemplaza al viejo `SERIAL`). No es nada que nosotros configuremos — es el comportamiento por defecto. **No hay CHECK constraints, no hay COMMENTS** a menos que los agreguemos explícitamente en el modelo.

### `backend/inventario/predictions.py` (106 líneas)

Algoritmo de promedio móvil simple (no necesita scikit-learn ni numpy):
1. Toma 90 días de ventas históricas
2. Calcula promedio móvil de 7 días (para suavizar)
3. Promedia los últimos 30 días para la proyección
4. Compara últimos 15 días vs 15 anteriores para tendencia
5. Calcula punto de reorden, cantidad sugerida, días hasta agotar, riesgo

## Flujo de autenticación (paso a paso)

```
FRONTEND                          BACKEND
────────                          ───────
1. Usuario llena login form
2. POST /api/auth/login/ ──────→ 3. Busca Usuario por correo
   {email, password}              4. Verifica password con check_password()
                                  5. Genera RefreshToken.for_user(user)
                                  6. Devuelve {access, refresh, user}
                                  ←──────────────────────────────
7. Guarda access_token y refresh_token en localStorage
8. Redirige a /dashboard
                                  9. Cualquier GET/POST a la API:
                                     Extrae token de Authorization header
                                     Valida con AccessToken(token_str)
                                     Obtiene usuario: Usuario.objects.get(id_usuario=token['user_id'])
                                  ←──────────────────────────────
10. Si la API responde 401:
    - api.ts interceptor borra tokens
    - Redirige a /login
11. Para export CSV/PDF:
    - El token se pasa como ?token=XXX en la URL
    - _obtener_usuario() lo extrae de query params
```

### ¿Por qué login manual en vez de SimpleJWT views?

Porque queremos devolver datos del usuario (`nombre`, `rol`, `negocio_id`) junto con los tokens en una sola llamada, sin hacer requests adicionales.

### Refresh de tokens

El frontend NO hace refresh automático actualmente. Si el access token expira (12h), el usuario recibe 401 y es redirigido al login. El refresh token (30 días) está almacenado en `localStorage` pero no se usa — es un area de mejora.

## Bugs conocidos y cómo arreglarlos

### 1. Reportes crashean al cambiar de pestaña

**Síntoma:** `Cannot read properties of undefined (reading 'map')` o `(reading 'toLocaleString')` en `ReporteInventario` o `ReporteMargenes`.

**Causa raíz:** React renderiza el nuevo componente con los `data` viejos de la otra pestaña antes de que el nuevo API request complete. `categorias_stock` e `ingresos_totales` no existen en los datos de ventas.

**Solución (3 capas):**
- Capa 1: `setData(null)` al cambiar pestaña o fecha → `!data ? null` previene render
- Capa 2: Flag `cancelled` en useEffect cleanup → ignora respuestas de requests cancelados
- Capa 3: `??` fallbacks en componentes → `data?.campo ?? valor_por_defecto`

**Archivos modificados:** `frontend/src/app/dashboard/reportes/page.tsx`

### 2. Search bar no funciona

**Síntoma:** Buscar productos por nombre no filtra.

**Causa raíz:** Faltaba `search_fields` en los ViewSets.

**Solución:** Agregar `search_fields = ['nombre', ...]` a ProductoViewSet, VentaViewSet, CompraViewSet, ProveedorViewSet.

**Archivos modificados:** `backend/inventario/views.py`

### 3. Compras vacías

**Síntoma:** La página de compras muestra "No hay datos disponibles".

**Causa raíz:** El seed data modificaba stock directamente sin crear objetos Compra/DetalleCompra.

**Solución:** Reemplazar el inline stock-adjust con `Compra.objects.create()` + `DetalleCompra.objects.create()`.

**Archivos modificados:** `backend/inventario/management/commands/seed_data.py`

### 4. Orden de ventas

**Síntoma:** Las ventas más viejas aparecen primero.

**Causa raíz:** No había `ordering` en los ViewSets. Django ordena por PK por defecto (ascendente).

**Solución:** `ordering = ['-fecha', '-id_venta']` en VentaViewSet y CompraViewSet.

**Archivos modificados:** `backend/inventario/views.py`

## Guía de modificaciones comunes

### Agregar un campo a Producto

1. Agregar el campo en `models.py`:
   ```python
   categoria = models.CharField(max_length=50, blank=True)
   ```
2. Crear migración: `python manage.py makemigrations && python manage.py migrate`
3. Si el frontend lo muestra, agregarlo al `ProductoSerializer` en `fields` y al type interface en el frontend.
4. Si se busca por ese campo, agregarlo a `search_fields` en el ViewSet.

### Agregar un nuevo reporte

1. Crear la vista function-based en `views.py` con `@api_view(['GET'])`
2. Agregar la ruta en `urls.py`
3. Crear el componente frontend en `reportes/page.tsx`
4. Agregar el tab en el array `tabs`

### Agregar una nueva regla de alerta

1. Editar `alerts.py` — agregar la condición después de las existentes
2. (Opcional) Agregar nuevo tipo a la lista `TIPOS` en el modelo `Alerta`
3. Migrar si se agregaron nuevos tipos

### Modificar el seed data

1. Editar `seed_data.py` — los datos están en arrays al inicio de cada sección
2. Re-ejecutar: `python manage.py seed_data`
3. Si hay datos existentes, resetear BD primero (ver sección abajo)

## Resumen de bugs encontrados y fixes aplicados

| Bug | Síntoma | Causa | Fix |
|-----|---------|-------|-----|
| Reportes crashean | `Cannot read properties of undefined` al cambiar pestaña | React renderiza componente nuevo con datos viejos | `setData(null)` en onChange + `cancelled` flag + `??` fallbacks |
| Search no funciona | Buscar productos no filtra | `search_fields` ausente en ViewSets | Agregar `search_fields` a ProductoViewSet y otros |
| Compras vacías | "No hay datos disponibles" en compras | Seed data no creaba objetos Compra | Reemplazar stock-adjust inline con `Compra.create()` |
| Orden ventas | Más viejas primero | Sin `ordering` en ViewSets | `ordering = ['-fecha', '-id_venta']` |
| Key warning en proveedores | "Each child should have unique key" | DataTable usa `item.id` pero serializador no exponía `id` | `id = serializers.IntegerField(source='id_proveedor')` |
| Sin email en proveedores | Campo email vacío en seed | Seed data no incluía email | Agregar email a `proveedores_data` + `update_or_create` |
| Sin detalle en compras | Solo mostraba conteo de productos | Columna usaba `c.detalles?.length` | Agregar `productos` (nombres concatenados) al serializer |
